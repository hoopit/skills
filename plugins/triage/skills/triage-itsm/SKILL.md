# Triage ITSM

Walk the open ITSM backlog and, per ticket, record a machine-readable verdict (custom fields) plus a
human-readable triage comment. Feeds the phase-2 fixer (`handle-jira-issue`), which consumes the
`AI: Agent Suitability` + score fields.

**Every comment posted during triage MUST start with this disclaimer:**

```
> *This was generated by AI during triage.*
```

## Reference files

- [RUBRIC.md](RUBRIC.md) — the classification taxonomy, scoring rubric, per-bucket behavior, and comment templates.
- [ATTACHMENTS.md](ATTACHMENTS.md) — how to download + analyze HAR files and screenshots (REST, not acli). Most ITSM tickets have these.
- [WRITE-RECIPE.md](WRITE-RECIPE.md) — `acli` read commands, auth, idempotency, and one-time field-creation recipe.
- [FIELD-MAP.md](FIELD-MAP.md) — human-readable `AI: <field>` → `customfield_XXXXX` → option-id reference (the live ids the scripts use are in `.claude/triage-config.json`).
- `workflow.js` — the **analyze-only** batch Workflow script (fans out one read-only agent per ticket → verdicts).
- `scripts/apply_verdicts.py` — the **deterministic writer**: takes a verdicts JSON file, sets the `AI:` fields
  (REST) + posts the comment (`acli`) + computes the score + stamps `Last Reviewed`. Supports `--dry-run`.
- Config: `.claude/triage-config.json` in the triage repo (org-level field/option ids + per-project blocks), created by the `setup-triage` skill — `apply_verdicts.py` reads it by default.
- `scripts/prepare_base_worktrees.sh` — fetch + refresh the read-only **base worktrees** the analysis
  runs against (pinned to each repo's latest default branch, fixed location, updated in place).

## Prerequisites

1. The central config exists — run **`setup-triage`** if `.claude/triage-config.json` is missing. (The
   `AI:` custom fields must already exist in Jira; see WRITE-RECIPE.md → *Field creation* for first-time setup.)
2. `JIRA_API_TOKEN` + `JIRA_EMAIL` are available (sourced from `~/.config/hoopit/jira.env`). Needed only
   for the REST field writes. Reads/comments use `acli` (verify `acli auth status`).

## Config (read, never hardcode)

All config comes from the central **`.claude/triage-config.json`** in this triage repo, created by the
`setup-triage` skill (run it first if the file is missing); `apply_verdicts.py` reads it directly. The
org-level keys used here: `jira_base_url` (`https://hoopit.atlassian.net`), `itsm_project` (`ITSM`), the
`AI:` `fields`/`options` ids, `valid_components`, and `status_map`. `AI: Area` → repo: `API`→api,
`Web Admin`→web-admin, `Flutter App`→flutter-app (these are also the keys of the config's `projects` map).

**Analysis runs against base worktrees, not the live repos.** The developer's checkouts may be on a
feature branch or dirty, which would skew root-cause analysis. So triage analyzes each repo at its
latest default branch via dedicated read-only worktrees (`scripts/prepare_base_worktrees.sh`,
default `$HOOPIT_ROOT/.triage-worktrees/<repo>`) — see the Prepare step below.

## Invocation

`/triage-itsm [--dry-run] [--limit N] [ITSM-123 ...]`

- **no args** — triage every ticket in `status = "Open"` (see Scope).
- **`--dry-run`** — analyze + report only; write **nothing** to Jira (no claim, no fields, no status change).
- **`--limit N`** — cap the number of tickets (useful for calibration).
- **explicit keys** — triage only those tickets.

## Scope (which tickets)

Only tickets in **`status = "Open"`** are triaged — status is the workflow gate:

```
project = ITSM AND status = "Open"
```

When support answers an `Additional information required` ticket they move it back to `Open`, so it
re-enters scope automatically on the next run. (`AI: Last Reviewed` is just an informational stamp now.)

## Flow

1. **Discover** — run the scope JQL, collect the `Open` keys. Honor `--limit` / explicit keys.
2. **Claim** (race-safety; skip on `--dry-run`) — transition the selected keys to **Under investigation**
   *before* analysis so a concurrent run won't re-grab them:
   ```bash
   acli jira workitem transition --key "ITSM-1,ITSM-2,..." --status "Under investigation" --yes
   ```
3. **Prepare base worktrees** — refresh the read-only checkouts the agents analyze against, so root
   cause is judged against the latest default branch (not whatever the dev has checked out):
   ```bash
   bash scripts/prepare_base_worktrees.sh   # prints name=path per repo
   ```
   Capture the `name=path` lines into a `repoPaths` map.
4. **Analyze (parallel)** — launch `workflow.js` (in this skill's directory) via the Workflow tool, passing `{ keys, skillDir,
   repoPaths }` as `args` (`skillDir` = this skill's own directory; `repoPaths` = the `name=path` map from step 3). One read-only `Explore`-type agent per ticket reads the ticket (+ linked
   `BAC`/`WEB`/`FA` issue + HAR/screenshots), does a lightweight root-cause hunt **in the base
   worktrees**, and returns a schema-validated verdict (incl. `components` and a deterministically-
   recomputed `priorityScore`). See RUBRIC.md.
5. **Apply** — save the workflow's returned verdicts array to a file, then run the writer:
   ```bash
   python3 scripts/apply_verdicts.py /tmp/itsm-verdicts.json   # add --dry-run to preview
   ```
   It sets the `AI:` fields (REST), sets `components` on Pending-development tickets, posts the triage
   comment (`acli`), stamps `Last Reviewed`, **and transitions each ticket to its target status**
   (Pending development / Additional information required / Escalated). See RUBRIC.md for the mapping.
6. **Report** — counts per suitability tier, the value×effort matrix, what went to `Additional
   information required`, and the ranked `Agent-ready` shortlist for phase-2.

If the Workflow tool isn't available/desired, process the keys sequentially in-context using the
per-ticket procedure in RUBRIC.md (claim first), then apply with the same script.

## Models & verification

- **Analyze agents:** `Explore` type pinned to **Sonnet** (`model: 'sonnet'` in `workflow.js`) — capable
  for code/HAR investigation + structured output, far cheaper than Opus across a backlog. Bump to
  `opus` there if a run needs deeper reasoning.
- **Orchestration** (discover/claim/apply/report) runs on the session model.
- **Apply** is a deterministic script — no model.
- **No semantic review stage.** One analyze agent per ticket; the verdict is applied after the
  deterministic guards only. To add adversarial verification (e.g. a second agent that tries to refute
  each `Agent-ready` before it routes to dev), add a second `workflow.js` stage that takes each verdict
  and returns a confirm/refute — at roughly double the analyze cost.

## Phase-2 handoff

Triaged ITSM tickets feed an autonomous fixer — the chain:

1. `triage-itsm` writes the verdict (AI fields + status + comment) on the ITSM ticket.
2. A **Jira automation** copies the `AI:` fields onto the linked platform issue (BAC/WEB/FA), keeping
   the dev board's verdict in sync automatically — no manual copy step.
3. **`/loop 30m /auto-fix-next-bug`** (api repo) burns down the BAC board: each run claims the highest
   `AI: Priority Score`, `Agent-ready`, open, unassigned BAC **bug** and runs `handle-jira-issue` in a
   worktree (branch → fix → regression test → CodeRabbit → PR), one at a time.
4. A human reviews and merges the PR.

Scope today is **BAC bugs only**. `Agent-assisted` / `Human-only` stay in Pending development for a
human; WEB/FA get their own fixer loops once BAC is proven.
