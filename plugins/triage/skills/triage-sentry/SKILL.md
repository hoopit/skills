# Triage Sentry

Runs as a single **cycle** — `review` then `promote` — on one fast loop. `review` is the LLM pass over the
unreviewed Sentry inbox; `promote` is a deterministic pass that feeds Jira within a WIP budget. **No local
state:** a fixable verdict is carried *on the Sentry issue itself* — `review` assigns it to the `ai-triage`
team and posts the brief/score as a comment; `promote` queries Sentry for that team and reads the verdict
back. The split stays *internal*: deferred-for-budget issues simply wait on Sentry (never re-analyzed), and
`promote` refills the board as the auto-fix loop closes bugs. Running `review` every ~30 min is cheap
because the `!assigned:#ai-triage` filter drops already-triaged issues at discovery — only *new* inbox
issues spawn agents (an empty inbox is a no-op), and `--limit` caps any spike. The two steps also run on
their own (`review` / `promote`).

## Invocation

- `/triage-sentry [--dry-run] [--limit N]` — **default: full cycle** (review → promote). Use on the loop.
- `/triage-sentry review [--dry-run] [--limit N] [<SENTRY-ID> ...]` — review step only.
- `/triage-sentry promote [--dry-run] [--max N]` — promote step only.

**Every Sentry note + Jira comment posted during triage MUST start with:**

```
> *This was generated by AI during triage.*
```

## Reference files

- [RUBRIC.md](RUBRIC.md) — disposition taxonomy, scoring, per-issue procedure, note/brief templates.
- [WRITE-RECIPE.md](WRITE-RECIPE.md) — Sentry CLI + Jira mechanics, native link, budget JQL, state files.
- `workflow.js` — the **review** analyze stage (one read-only `Explore` agent per issue → verdicts) plus a
  **Reconcile** phase that converges same-signature clusters that disagreed on disposition (see *Reconcile*).
- `scripts/apply_review.py` — the **review** writer (Sentry actions, escalation Jira+link, and for
  Develop/Silence: assign `ai-triage` + post the verdict as a comment).
- `scripts/promote_pending.py` — the **promote** step (budget → query Sentry `ai-triage` work-list → read
  verdict comment → BAC bug + AI fields + native link).
- Config: `.claude/triage-config.json` in the triage repo (org-level ids + per-project Sentry/Jira blocks), created by `setup-triage`. The writers read it by default; pick a project with `--project <key>` (default `api`).

## Config (read, never hardcode)

All config comes from the central **`.claude/triage-config.json`** in this triage repo, created by the
`setup-triage` skill (run it first if missing). Org-level keys: `sentry_org` (`hoopit`), `jira_base_url`,
the `AI:` `fields`/`options`, labels, budget. **Per-project** values live under `projects.<key>`
(`sentry_project`, `sentry_project_id`, `jira_project`, `default_area`, `default_component`); the writers
take `--project <key>` (default `api`) and merge that block over the org-level keys. Today only `api`
(→ Sentry `bac`, Jira `BAC`) has its Sentry ids filled; `web-admin` / `flutter-app` need their Sentry slug
+ numeric id added (via `setup-triage`) before they can be triaged here.

## Prerequisites

1. `sentry auth status` shows the `hoopit` org (the CLI's stored token drives all Sentry reads/writes).
2. `acli auth status` OK; `JIRA_API_TOKEN` + `JIRA_EMAIL` available (from `~/.config/hoopit/jira.env`) for
   REST field writes.
3. The **`ai-triage`** Sentry team exists **and has the `bac` project** (`POST projects/hoopit/bac/teams/
   ai-triage/`). This is load-bearing: Sentry's `assigned:#ai-triage` search only matches issues in the
   team's projects, so without the project link `promote`'s work-list query returns nothing even though the
   assignment succeeds. Keep `triage_assignee` in the config (`team:ai-triage`) matching its slug.
4. The `AI:` fields exist in Jira (they do — shared with triage-itsm).

Analysis runs against a read-only **base worktree** of `api` (latest default branch), refreshed by the
sibling `triage-itsm` skill's `scripts/prepare_base_worktrees.sh` — not the developer's possibly-dirty checkout.

## Default: full cycle (review → promote)

With no mode argument, run both steps in sequence in one turn — this is what the loop uses:

1. **Review** — run the review step below (discover → prepare worktree → `workflow.js` → `apply_review.py`).
   If discovery finds no new inbox issues, skip straight to promote.
2. **Promote** — run `promote_pending.py` to create Jira bugs for `ai-triage`-assigned issues within budget.
3. **Report** — one combined summary: review counts per disposition, then promote's promoted/headroom.

Pass `--limit N` (default 10 for the cycle) to cap how many inbox issues `review` analyzes per tick;
`--dry-run` propagates to both steps. Then the individual steps:

## Mode: `review` (LLM)

`/triage-sentry review [--dry-run] [--limit N] [<SENTRY-ID> ...]`

- **no ids** — review the For-Review inbox (see Scope). **`--dry-run`** — analyze + report only; no Sentry
  or Jira writes, nothing queued. **`--limit N`** — cap issues (calibration). **explicit ids** — only those.

### Scope (which issues)

Sentry's **For Review** inbox is the untriaged set (analogue of ITSM `status = Open`). Discover via the
Sentry MCP `search_issues` (org `hoopit`, project `bac`):

```
is:unresolved is:for_review environment:production !assigned:#ai-triage      (sort: date — last seen, limit N)
```

The `environment` filter and sort order are config in `.claude/triage-config.json` (`sentry_environment` = `production`,
`discover_sort` = `date`). **Scope is production only**, and we **sort by `date` (last seen), not frequency**:
actively-occurring issues are reviewed first and stale/dormant ones aren't pulled ahead of them — high-volume
issues are usually also recently-seen, so they still surface quickly under a `--limit`.

Dedup is **purely Sentry-side, no local state:** `review` assigns every Develop/Escalate/Silence issue to
the `ai-triage` team (which always exists), and Archive/Resolve leave `is:unresolved` — so
`!assigned:#ai-triage` excludes everything already handled, even across machines and after a wiped
checkout, and even when an issue stays in For Review post-assignment (ownership rules don't clear it). The
team slug comes from `triage_assignee` in the config (`team:ai-triage` → `#ai-triage`). An issue
Sentry now flags `is:regressed`/`is:escalating` is allowed back in (a previously archived error that comes
back gets re-triaged) — Archive/Resolve issues aren't assigned to the team, so the filter doesn't block
their return. Capture each issue's **short id + numeric id** (search results include both).

### Flow

1. **Discover** — run the scope query (Sentry MCP `search_issues`, incl. `!assigned:#ai-triage`);
   collect `{shortId, numericId, title}`. Honor `--limit` / explicit ids. (The `!assigned:#ai-triage`
   filter is the only dedup — no ledger lookup.)
2. **Prepare base worktree** — `bash ../triage-itsm/scripts/prepare_base_worktrees.sh`
   (the sibling `triage-itsm` skill owns it) and capture the `api=<path>` line into `repoPaths`.
3. **Analyze + Reconcile (parallel)** — launch `workflow.js` (in this skill's directory) via the Workflow tool with
   `{ issues, skillDir, repoPaths }` (`skillDir` = this skill's own directory). One read-only `Explore`/`sonnet` agent per issue → schema-validated
   verdict (see RUBRIC.md), then a built-in **Reconcile** phase harmonizes same-signature clusters (see
   *Reconcile* below). The workflow returns the **post-reconcile** verdicts — save those for Apply. If the
   Workflow tool isn't available, process issues sequentially in-context using the RUBRIC procedure (and
   reconcile any same-signature conflicts by hand per *Reconcile* below).
4. **Apply** — save the (reconciled) verdicts array to a file, then run the writer:
   ```bash
   python3 scripts/apply_review.py /tmp/sentry-verdicts.json   # add --dry-run to preview
   ```
   It applies the Sentry action per disposition (archive-until-escalating / resolve), creates Jira **only
   for Escalate** (cap-exempt) with a native link, and for **Develop/Silence-in-code** assigns the issue to
   `ai-triage` and **posts the verdict (brief + score) as a Sentry comment** (Silence is also archived
   forever). No local queue. (Appends an audit line to `triaged.log` — not used for control flow.)
5. **Report** — counts per disposition, value×effort matrix, and what was archived/resolved/escalated.
   Call out any cluster the Reconcile phase converged or escalated.

### Reconcile (auto-harmonize same-signature clusters)

Because each inbox issue is analyzed by an **independent** agent, several Sentry issues that share one root
cause (the same exception/message raised from different endpoints/culprits) can land **inconsistently** —
e.g. the same broker outage scored `Resolve` on two issues and `Escalate` on a third, or **unanimously
wrong** (every issue of an already-fixed incident independently marked `Escalate`). The orchestrator should
**not** punt these to a human by default; `workflow.js` handles them automatically in a **Reconcile** phase
that runs right after Analyze (zero cost when there are no multi-member clusters — grouping is pure JS, no
agent):

1. **Group** verdicts by a normalized title key (digits → `#`, punctuation stripped) so same-signature
   issues cluster even when ids/counts differ.
2. **Select clusters** — **every** cluster with ≥2 members is reconciled, *not only* ones whose members
   disagree. Triggering only on disagreement would miss a **unanimous-but-wrong** cluster (no internal
   conflict to detect). Singletons pass through untouched.
3. **Reconcile** — one read-only `Explore`/`sonnet` agent per cluster re-investigates the **shared**
   evidence (per-member `lastSeen`/`substatus`/event timeline + `git log` around the last-seen date for a
   fix/deploy + a **cross-batch search** for sibling issues with the same signature already resolved or
   assigned to `ai-triage` in earlier runs) and returns one of:
   - **converged** → all members get the agreed disposition (it may *overrule* a unanimous-but-wrong verdict,
     e.g. flipping `Escalate`→`Resolve` for an already-fixed incident); the workflow rebuilds each member's
     `sentryNote` and (for Develop/Escalate/Silence) per-member Jira fields, so Apply stays consistent.
   - **keep-split** → the members are genuinely different (e.g. one endpoint still erroring post-fix); their
     independent verdicts are kept.
   - **cannot-converge** → the **whole cluster is escalated to a human** (the safe, loud fallback) — never
     silently dropped.

The phase only edits clustered verdicts; singletons are unchanged. Apply/promote consume the reconciled
verdicts unchanged — no script changes. **Note:** Reconcile only sees same-*title* clusters; same-root-cause
issues with *different* titles (e.g. one payment bug surfacing under two messages) won't auto-cluster — the
analyze stage records each issue's culprit verbatim to help, but cross-title dedup before promote is still a
manual/future step.

## Mode: `promote` (deterministic, zero-LLM)

`/triage-sentry promote [--dry-run] [--max N]` — pure script; safe on a fast loop.

```bash
python3 scripts/promote_pending.py            # or --dry-run / --max N
```

Computes Jira headroom (`15 − unresolved BAC issues labelled sentry but not sentry-escalated`; counts by
`resolution IS EMPTY`, so Awaiting-release/Done don't consume budget), then queries Sentry for the
**work-list** (`assigned:#ai-triage !is:resolved` — covers Develop=unresolved and Silence=ignored). For
each, it skips any already natively linked (already promoted/escalated) or carrying no verdict comment,
reads the verdict from the issue's comment, ranks by `priorityScore`, and for each up to headroom: dedups
Jira, creates the BAC **Bug** + `AI:` fields + labels, native-links it from Sentry, and posts a Sentry
note. Moving an issue off `ai-triage` drops it from the work-list (the veto); `review` (scoped
`!assigned:#ai-triage`) re-picks-it-up — the two scopes are exact complements, so nothing strands. As
`auto-fix-next-bug` closes bugs and frees budget, the next run refills it — no re-analysis.

## Phase-2 handoff (no new code)

`promote`-created `Agent-ready` BAC bugs (with `AI:` fields + `sentry` label, status `Open`) are picked up
by the **existing** `/loop 30m /auto-fix-next-bug` — its selector orders by `AI: Priority Score` and
filters `AI: Agent Suitability = Agent-ready`, with no label restriction. `Agent-assisted`/`Human-only`
Develop bugs and `Escalate` issues wait for a human.

## Cadence

```bash
/loop 30m /triage-sentry            # full cycle: review the new inbox, then promote within budget
```

One loop does both steps every ~30 min: `review` analyzes only genuinely new inbox issues (deduped by the
`!assigned:#ai-triage` filter,
`--limit`-capped, no-op when the inbox is clear), then `promote` refills the Jira board as `auto-fix-next-bug`
frees room. Calibrate first with `/triage-sentry --dry-run --limit 5` (and, on a large initial backlog, a
few bounded manual `review` runs) before enabling the loop.

## Models & verification

- **Analyze agents:** `Explore` type pinned to **Sonnet** (`workflow.js`) — capable for code/stacktrace
  investigation + structured output, far cheaper than Opus across a backlog.
- **Apply (review)** and **promote** are deterministic scripts — no model.
- **No semantic review stage.** To add adversarial verification (a second agent that tries to refute each
  `Develop/Agent-ready` before it's posted), add a second `workflow.js` stage at ~double the analyze cost.
