# Write Recipe — Sentry (CLI) + Jira (acli + REST)

> **The canonical writers are [`scripts/apply_review.py`](scripts/apply_review.py) (review side) and
> [`scripts/promote_pending.py`](scripts/promote_pending.py) (promote side).** Don't hand-roll writes —
> feed `apply_review.py` a verdicts JSON file (`--dry-run` to preview). The recipes below document the
> mechanics they implement, for debugging.

## Auth

- **Sentry** — the authenticated `sentry` CLI handles everything; **no token env var**. Confirm with
  `sentry auth status` (must show the `hoopit` org). `sentry api` is a REST proxy (endpoint relative to
  `/api/0/`); `sentry issue archive|resolve` are first-class status commands.
- **Jira** — reads/comments/transitions/create via `acli` (OAuth; `acli auth status`). Custom-field /
  label / priority writes via the Jira REST API with `JIRA_EMAIL` + `JIRA_API_TOKEN` (from
  `~/.config/hoopit/jira.env`). AI: field ids live in the central `.claude/triage-config.json`
  (managed by `setup-triage`; global Jira contexts shared with BAC/WEB/FA).

## Sentry status changes (first-class commands)

```bash
sentry issue archive <shortId> --until auto   # Archive — re-opens on a spike ("until escalating")
sentry issue archive <shortId>                # Silence-in-code — archive forever
sentry issue resolve <shortId>                # Resolve
```

## Sentry assign / priority / note / native link (`sentry api`)

```bash
# priority (reliable) and assignment (best-effort — needs the team to exist)
sentry api issues/<numericId>/ -X PUT -d '{"priority":"high"}'
sentry api issues/<numericId>/ -X PUT -d '{"assignedTo":"team:ai-triage"}'

# activity-feed note (starts with the AI disclaimer)
sentry api issues/<numericId>/comments/ -X POST -d '{"text":"> *This was generated by AI during triage.* ..."}'

# native two-way Jira link of an ALREADY-created Jira issue (the UI's "Link Jira Issue")
sentry api groups/<numericId>/integrations/12493/ -X PUT -d '{"externalIssue":"BAC-1234"}'
```

- `numericId` is the Sentry group id (`id` from `sentry issue view --json`), **not** the short id.
- **Jira integration id = `12493`** (in `.claude/triage-config.json`; re-derive with
  `sentry api 'organizations/hoopit/integrations/?provider_key=jira'`). The integration has `issue-sync`
  enabled, so the native link is two-way.
- Use `sentry api ... --dry-run` (`-n`) to preview the resolved request without sending it.
- A text reference (e.g. the Sentry short id in a Jira comment) does **not** create the native link — only
  the `groups/.../integrations/...` `PUT` does. If it fails, the writer falls back to a Jira comment +
  Sentry note carrying the URLs, and logs the degradation.

## Jira: dedup → create → fields/labels (BAC has no Pending-development/Escalated status)

```bash
# dedup (the writer also checks the Sentry issue's native annotations first)
acli jira workitem search --jql 'project = BAC AND text ~ "<shortId>"' --fields 'key,status' --csv

# create a Bug (writer parses the BAC-#### key from stdout)
acli jira workitem create --project BAC --type Bug --summary '<title>' --description '<desc with ## References - Sentry: <shortId>>'

# AI: fields + components + priority via REST; labels via update-ops (don't clobber existing)
curl -sf -X PUT -u "$JIRA_EMAIL:$JIRA_API_TOKEN" -H 'Content-Type: application/json' \
  "https://hoopit.atlassian.net/rest/api/3/issue/BAC-1234" --data '{
    "fields": {
      "customfield_10975": {"id":"10846"},   # AI: Agent Suitability (Agent-ready)
      "customfield_10976": {"id":"10851"},   # AI: Value (High)
      "customfield_10977": {"id":"10856"},   # AI: Effort (Low)
      "customfield_10978": {"id":"10857"},   # AI: Confidence (High)
      "customfield_10979": {"id":"10860"},   # AI: Area (API)
      "customfield_10980": 300,               # AI: Priority Score
      "customfield_10981": "<now>",           # AI: Last Reviewed
      "components": [{"name":"Backend"}]
    },
    "update": { "labels": [{"add":"sentry"}] }
  }'   # 204 on success
```

**Routing (no status transitions needed):** newly-created BAC bugs land in **Open**, which is exactly
what `auto-fix-next-bug` selects. Differentiation is by AI: Agent Suitability + labels + priority:
- **Develop** keeps its verdict suitability; only `Agent-ready` is auto-fixed. Label `sentry`.
- **Silence-in-code** is `Agent-ready`. Labels `sentry` + `sentry-suppression`.
- **Escalate** is forced to `Human-only` (so the fixer's `= Agent-ready` filter skips it), High priority,
  labels `sentry` + `sentry-escalated`; a transition to `Escalated` is attempted best-effort and skipped
  if no such status exists.

## Budget (cap 15, escalations exempt)

```bash
acli jira workitem search --csv --paginate --fields key \
  --jql 'project = BAC AND labels = sentry AND labels != sentry-escalated AND resolution IS EMPTY'
```
`headroom = 15 - count`. `labels != sentry-escalated` keeps escalations out of the count. **`resolution IS
EMPTY`** (not `statusCategory != Done`) is the soundness fix: an issue with a resolution is completed/ok —
this counts Done *and* "Awaiting release/deployment" (which keep an In-Progress statusCategory but are
functionally finished) as not consuming the active WIP budget.

## State lives on the Sentry issue (no local queue)

For a Develop/Silence-in-code issue, `review` carries the verdict **on the issue itself** rather than in a
local file: it assigns the issue to `ai-triage` and posts a comment whose human note ends with a
marker-delimited one-line JSON the `promote` step parses back:

```
> *This was generated by AI during triage.* … human-readable note …

===AI-TRIAGE-VERDICT-V1===
{"disposition":"Develop","agentSuitability":"Agent-ready","value":"High","effort":"Low","confidence":"High","area":"API","priorityScore":300,"jiraSummary":"…","jiraDescription":"…","jiraBrief":"…"}
```

`promote` reads it with `GET issues/<numericId>/comments/` and takes the **latest** verdict comment (so a
re-review supersedes an older one). The only local file is `…/.claude/local/triage-sentry/triaged.log`, an
**append-only audit log** — nothing reads it for control flow; safe to delete/rotate.

> Note: the skill never deletes notes (and the CLI token can't — delete needs a higher scope), so
> re-reviewing an issue *appends* a newer verdict comment. *Latest wins*, so a stale comment never affects
> correctness — it's only visual clutter on a re-reviewed issue (uncommon), and a human can delete old ones
> in the Sentry UI if desired.

## Idempotency & prerequisites

- **Dedup is purely Sentry-side (no local state).** `review` discovers `!assigned:#ai-triage`; it assigns
  every Develop/Escalate/Silence issue to `ai-triage`, and Archive/Resolve leave `is:unresolved`, so the
  query excludes everything already handled — across machines and after a wiped checkout, and even when an
  issue stays in For Review post-assignment (ownership rules don't clear it). An issue Sentry now flags
  `is:regressed`/`is:escalating` is allowed back in (Archive/Resolve issues aren't assigned to the team, so
  a re-spiked error returns).
- **`promote` is driven by the live `ai-triage` work-list.** It queries `assigned:#ai-triage !is:resolved`
  (covers Develop=unresolved and Silence=ignored), and per issue **skips** anything already natively linked
  (already promoted/escalated) or with no verdict comment. Moving an issue off `ai-triage` drops it from the
  work-list — that's the veto: `review` (scoped `!assigned:#ai-triage`) re-picks-it-up and re-triages. So
  `review` (`!assigned`) and `promote` (`assigned`) are exact complements — no gaps, nothing stranded. (To
  permanently stop an issue, resolve/ignore it — moving it off the team only forces a re-triage.)
- **`team:ai-triage`** exists **and has the `bac` project** — it's the "reviewed" marker, the dedup filter,
  **and** promote's work-list. Load-bearing: `assigned:#ai-triage` search only matches issues in the team's
  projects, so without `POST projects/hoopit/bac/teams/ai-triage/` the assignment lands but the work-list
  query returns nothing. Keep `triage_assignee` in the config (`team:ai-triage`) matching the slug.
  (Required Sentry token scopes are just assign/comment/archive/resolve/link/priority — all confirmed; the
  skill never deletes comments or unlinks, so it needs no admin/delete scope.)
- An **`Escalated`** BAC status is optional — the escalate transition is best-effort.
